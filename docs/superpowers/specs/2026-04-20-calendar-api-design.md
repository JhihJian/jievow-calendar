# 日期/阴历/节气 API 系统设计

## 概述

一个公开、免费、无需认证的 API 服务，输入公历日期，返回对应的阴历信息和节气信息。未来可通过字段选择参数获取节气补充信息（描述、推荐食物、养生建议等）。

### 约束

- 支持日期范围：2025-01-01 至 2027-12-31
- 节气精度：仅返回日期，不返回精确时刻
- 节气时区：北京时间（UTC+8）
- 完全开放，无认证/鉴权
- 调用方不限（Web 前端、后端服务、移动端等）

---

## 方案选型

选择**数据表驱动**方案：预先生成日期对照数据，API 查表返回。

理由：
1. 日期范围有限（3 年约 1096 条记录），数据量小
2. 查表即返回，性能和稳定性最优
3. 避免自行实现复杂的阴历/天文算法
4. 扩展新字段只需在数据中增加列

---

## 数据模型

核心实体：日期记录，每条对应一个公历日期。

### 主表：日历记录

| 字段 | 类型 | 说明 |
|------|------|------|
| `date` | string | 公历日期，ISO 8601 格式（如 `"2026-04-20"`），主键 |
| `lunar_year` | number | 农历年 |
| `lunar_month` | number | 农历月（1-12） |
| `lunar_day` | number | 农历日（1-30） |
| `is_leap_month` | boolean | 当前月是否为闰月 |
| `lunar_year_ganzhi` | string | 年干支（如 `"丙午"`） |
| `lunar_month_ganzhi` | string | 月干支 |
| `lunar_day_ganzhi` | string | 日干支 |
| `solar_term` | string \| null | 节气名称，非节气日为 null |

### 扩展表：节气补充信息（未来）

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | string | 节气名称，主键（共 24 条） |
| `description` | string | 节气简介 |
| `recommended_foods` | string[] | 推荐食物列表 |
| `health_tips` | string | 养生建议 |

查询时按 `solar_term` 字段关联补充信息。

---

## API 接口定义

### 端点

```
GET /api/v1/date/{date}?fields=basic,lunar,solar_term,supplement
```

### 路径参数

| 参数 | 格式 | 说明 |
|------|------|------|
| `date` | `YYYY-MM-DD` | 查询的公历日期 |

### 查询参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `fields` | `basic,lunar,solar_term` | 逗号分隔的字段组 |

**所有响应都包含 `data_version` 字段**（如 `"2025a"`），标识数据版本，便于排查问题。

**字段组定义：**

| 字段组 | 返回内容 |
|--------|----------|
| `basic` | 公历日期、星期几 |
| `lunar` | 阴历年月日、闰月标记、干支 |
| `solar_term` | 节气名称（非节气日为 null） |
| `supplement` | 节气补充信息（需 `solar_term` 非空时有值，否则 null） |

### 响应示例

**节气日：**

```json
{
  "data_version": "2025a",
  "date": "2026-04-20",
  "weekday": "星期一",
  "lunar": {
    "year": 2026,
    "month": 3,
    "day": 3,
    "is_leap_month": false,
    "year_ganzhi": "丙午",
    "month_ganzhi": "壬辰",
    "day_ganzhi": "甲寅"
  },
  "solar_term": {
    "name": "谷雨"
  }
}
```

**非节气日：**

```json
{
  "data_version": "2025a",
  "date": "2026-04-21",
  "weekday": "星期二",
  "lunar": {
    "year": 2026,
    "month": 3,
    "day": 4,
    "is_leap_month": false,
    "year_ganzhi": "丙午",
    "month_ganzhi": "壬辰",
    "day_ganzhi": "乙卯"
  },
  "solar_term": null
}
```

**带补充信息（未来扩展）：**

```json
{
  "data_version": "2025a",
  "date": "2026-04-20",
  "weekday": "星期一",
  "lunar": { "..." : "..." },
  "solar_term": {
    "name": "谷雨"
  },
  "supplement": {
    "description": "春季最后一个节气，雨生百谷",
    "recommended_foods": ["香椿", "春笋", "菠菜"],
    "health_tips": "宜健脾祛湿，多食甘味"
  }
}
```

### 错误响应

| 状态码 | 场景 | 响应体 |
|--------|------|--------|
| 400 | 日期格式无效 | `{"error": "invalid_date", "message": "日期格式应为 YYYY-MM-DD"}` |
| 400 | 未知 fields 值 | `{"error": "invalid_fields", "message": "合法值: basic, lunar, solar_term, supplement"}` |
| 404 | 日期超出支持范围 | `{"error": "date_out_of_range", "message": "支持的日期范围: 2025-01-01 至 2027-12-31"}` |

## 数据来源与生成

### 数据来源

- **阴历数据**：使用成熟的开源阴历库/数据表，离线生成 2025-2027 年的每日对照记录
- **节气数据**：从天文算法库或权威数据源获取，共约 72 条记录（3 年 × 24 节气）

### 数据校验

生成后抽样与在线万年历（香港天文台、中科院紫金山天文台公开数据）交叉验证。

### 生成流程

```
权威数据源 → 生成脚本 → 结构化数据文件 → API 启动时加载到内存
```

- 生成脚本独立运行，与 API 服务解耦
- 数据文件纳入版本控制，可审计、可回滚
- API 启动时校验数据文件 checksum

### 补充信息维护

- 24 条节气补充记录，按节气名称索引
- 手动编写或从资料源导入，单独文件维护

## 系统结构

```
┌──────────────────────────────────────┐
│           公开 API 服务               │
│                                      │
│  请求路由 → 查询引擎（内存查表）      │
│                  ↓                   │
│          预加载数据文件               │
│        (阴历+节气+补充信息)           │
└──────────────────────────────────────┘

┌──────────────────────────────────────┐
│        离线数据生成工具               │
│  权威数据源 → 生成脚本 → 数据文件     │
└──────────────────────────────────────┘
```

### 运行时特点

- **无外部依赖**：不依赖数据库、缓存等服务
- **冷启动即可用**：加载数据文件后立即对外服务
- **无状态**：适合水平扩展

### 版本策略

- API 路径含版本号（`/api/v1/`），字段变更时升版本
- 数据文件随版本发布，响应含 `data_version` 字段便于排查
- 新增 fields 值不影响旧调用方，向后兼容

## 测试策略

### 数据准确性测试

- 准备 20-30 个基准日期（覆盖闰月、节气日、除夕、春节、月末等边界）
- 每次数据生成后自动与基准数据比对，不一致则阻断发布

### API 功能测试

- 正常查询：有效日期 + 各种 fields 组合
- 错误输入：格式错误、超出范围、未知 fields
- 节气日 vs 非节气日的返回结构差异

### 质量保障

- 数据文件有 checksum，API 启动时校验
- 响应含 `data_version` 字段，便于问题排查
